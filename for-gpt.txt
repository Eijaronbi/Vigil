# VIGIL — Complete Project Context for AI

## What We Are Building

Vigil is a multi-platform message monitoring system that:
- Listens across WhatsApp, Telegram, Twitter/X, and Gmail
- Classifies messages by importance using AI (keyword + LLM scoring)
- Reads important messages aloud via TTS (edge-tts on PC, TTS on Android)
- Attests critical messages on-chain (Monad testnet)
- Works with screen off on Android (foreground service + notification listener)

Target users: power-users who are in many groups/channels and need hands-free awareness without checking their phone constantly.

## Architecture

```
User's MetaMask ──sign auth──> Backend ──session token──> Frontend (Dashboard)

User's PC Browser / Chrome Extension ──┐
                                       ├──> FastAPI Backend (port 8002) ──> SQLite DB
User's Android Phone ──────────────────┘         │
          ┌───────────────────────────────────────┤
          │                                       ├──> Telegram Bot API (polling)
          │                                       ├──> Gmail OAuth 2.0 (IMAP)
          │                                       ├──> Twitter/X (Jina Reader polling)
          │                                       ├──> WebSocket ──> Dashboard (real-time alerts)
          │                                       └──> MemoryVault Contract (Monad)
          │                                                 │
          │                                          Merkle Root (SHA-256 of batch)
          │                                                 │
          │                                          Monad Block (immutable, chain 10143)
```

## WHAT HAS BEEN DONE

### Backend (FastAPI — hackathon/backend/)
- Full project structure with routers, models, middleware, config
- **Auth router** (`routers/auth.py`): POST /api/auth/login (password → Bearer token, HMAC-signed, 24h expiry), POST /api/auth/logout, GET /api/auth/verify
- **Sources router** (`routers/sources.py`): GET /api/sources (list all sources + status), POST /api/sources/telegram/connect (verify token + start polling), POST /api/sources/telegram/disconnect, GET /api/sources/telegram/info, GET /api/sources/telegram/groups
- **Messages router** (`routers/messages.py`): GET /api/messages (query with source/group_name/important filters, paginated), POST /api/messages (create + auto-classify)
- **Dashboard router** (`routers/dashboard.py`): serves index.html at GET /
- **Security middleware** (`middleware.py`): SecurityHeadersMiddleware (CSP, HSTS, X-Frame-Options, etc.) + RateLimitMiddleware (60 req/min/IP, returns 429 with Retry-After header)
- **Global exception handler**: catches all unhandled exceptions, returns 500 with request_id
- **CORS**: locked to localhost:8002 and localhost:8080
- **Database**: SQLite via SQLAlchemy 2.0, auto-creates tables + seeds default user (id=1) on startup
- **Models**: User, Group, Message, Rule, DigestQueue, DailyReport
- **WebSocket manager** (`websocket_manager.py`): singleton WebSocketManager — connect/disconnect/broadcast. All connected dashboards receive real-time alerts
- **Telegram watcher** (`watchers/telegram_watcher.py`): uses python-telegram-bot (Application + MessageHandler + polling). When a message arrives in a monitored group, it calls back into sources.py which saves to DB + broadcasts via WS
- **TTS engine** (`tts.py`): edge-tts (Microsoft Edge TTS), falls back to browser SpeechSynthesis if server unavailable
- **Config** (`config.py`): pydantic-settings loading from .env — telegram_bot_token, gmail_oauth_*, groq_api_key, openrouter_api_key, monad_rpc_url, auth_password, etc.

### Frontend — Website Hub (hackathon/index.html)
- Single HTML file served by FastAPI at /
- Kombai terminal aesthetic design
- **Sections**: Hero, How It Works (4 steps), Downloads (Chrome Extension + Android), Sources (6 cards), Dashboard (login → source connection cards + message feed + search), Voice Demo (TTS test), CTA, Footer
- **Auth flow**: password input → POST /api/auth/login → stores token in JS var → shows session panel
- **Source connection cards**: Telegram (prompt for bot token → POST /api/sources/telegram/connect → shows groups), Discord (prompt for bot token → connects discord.py watcher → shows servers), Twitter/X (add handles → Jina Reader polls profile), Facebook (add handles → Jina Reader polls page), Gmail (shows setup instructions), WhatsApp (download extension)
- **Message feed**: GET /api/messages with pagination, search bar supports "source:telegram" and "sender:name" queries
- **WebSocket**: connects to ws://localhost:8002/ws, auto-reconnects every 5s, refreshes message list on alert
- **TTS demo**: type text → GET /api/tts?text=... → plays audio (edge-tts) or falls back to browser SpeechSynthesis
- **Error states**: login error message shown inline (not alert()), message fetch errors shown inline, loading spinner states

## Authentication — Two Methods

### Method 1: Password Login (Fallback)
- **Endpoint**: POST /api/auth/login with JSON `{"password": "vigil"}`
- Same Bearer token system as wallet login

### Method 2: Wallet Sign-In (EIP-4361 style)
- **Step 1**: Frontend calls POST /api/auth/nonce with `{"address": "0x..."}` → gets a unique hex nonce (expires 300s)
- **Step 2**: User signs message `"Vigil\nAddress: {address}\nNonce: {nonce}"` in MetaMask using `personal_sign`
- **Step 3**: Frontend sends POST /api/auth/wallet-login with `{address, signature, nonce}` → gets Bearer token
- **Backend verification**: Uses `eth_account.Account.recover_message()` to recover signer address, compares with claimed address, pops nonce to prevent replay
- **Token format**: `vgl_{48-char-random}_{12-char-HMAC-SHA256-signature}`
- **Expiry**: 24 hours from creation
- **Storage**: in-memory dict (tokens lost on server restart, wallet address stored with token)

### Frontend
- Uses ethers.js v6 (CDN) for wallet interaction
- Token stored in JavaScript variable (not localStorage — page refresh loses it)
- Session status bar shows wallet address + network badge
- Memory Vault panel visible only when connected via wallet
- Network switcher (testnet/mainnet) calls `wallet_switchEthereumChain`

## Source-Specific Details

### WhatsApp
- **PC**: Chrome Extension reads web.whatsapp.com DOM directly — mimics human reading, no protocol hacking, no ban risk. Users navigate to web.whatsapp.com, scan QR code normally, extension reads messages from DOM.
- **Android**: Notification Listener Service captures WhatsApp notifications in real time. One permission grant, zero ToS risk.
- **Status**: Chrome extension scaffold exists in hackathon/extension/. Android scaffold in hackathon/android-app/. Both need further testing.

### Telegram
- **Single bot**: Developer creates ONE bot (@vigil_bot) via @BotFather, gets token, puts in .env or dashboard UI
- **Usage**: Users add @vigil_bot to any Telegram group they want monitored
- **Protocol**: Bot API polling (no webhooks — simpler for local dev). Uses python-telegram-bot library.
- **Watcher**: Background polling via telegram.ext.Application. When message arrives → saves to DB + broadcasts via WebSocket to all dashboard clients.
- **Connect flow**: Click Telegram card → prompt for token → POST /api/sources/telegram/connect → server verifies token via bot.get_me() → starts polling → shows connected groups below cards

### Twitter/X (Jina Reader)
- Polls public profile pages via r.jina.ai every 2 minutes
- Parses timestamps + text from markdown response
- No auth needed — public content only
- Endpoints: POST /api/sources/jina/connect, /disconnect, GET /profiles
- Configurable handles, stores last content to diff against

### Gmail
- **Protocol**: OAuth 2.0 with Gmail API
- **Setup required**: Developer must create Google Cloud project, enable Gmail API, create OAuth credentials, set redirect URI to http://localhost:8002/api/sources/gmail/callback
- **Config vars**: GMAIL_OAUTH_CLIENT_ID, GMAIL_OAUTH_CLIENT_SECRET, GMAIL_OAUTH_REFRESH_TOKEN in .env
- **Status**: Dashboard shows setup instructions on connect click. Backend ready to accept tokens but OAuth flow not fully implemented.

### Discord
- Bot setup: create app at discord.com/developers, enable Message Content Intent, invite to server
- Watcher: discord.py with Gateway intents, real-time via on_message event
- Endpoints: POST /api/sources/discord/connect, /disconnect, GET /info (servers list)
- Same save+broadcast pattern as Telegram

### Facebook (Jina Reader)
- Same Jina Reader pattern as Twitter
- Polls public Facebook pages
- Parses long-form posts from markdown
- Groups not accessible (privacy restrictions)

## Port Situation

- **Primary port**: 8002 (works correctly)
- **Port 8080 ghost zombie**: Windows TCP bind won't release. A process bound port 8080 but `netstat -ano | findstr :8080` shows no PID. It's a Windows kernel-level TCP bind leak. Port 8080 is unusable on this machine until next reboot.
- **Port 8000**: Also has ghost zombie. Same issue.
- **All files updated**: Every reference to 8000 or 8080 changed to 8002.

## Memory Vault — On-Chain Message Commitments

### Smart Contract — MemoryVault.sol
- **Location**: `contracts/contracts/MemoryVault.sol`
- **Network**: Monad Testnet (chain ID 10143), Mainnet (chain ID 143)
- **Solidity**: 0.8.27, EVM version Prague, with optimizer
- **Functions**:
  - `commitBatch(bytes32 merkleRoot, uint256 messageCount, bytes32 metadataHash)` → batchId — stores a Merkle root on-chain. Emits `BatchCommitted` event.
  - `verifyMessage(bytes32 merkleRoot, bytes32 leaf, bytes32[] proof)` → bool — verify a message was in a committed batch via Merkle proof
  - `getBatch(uint256 batchId)` → Batch struct (root, timestamp, count, metadataHash, committer)
  - `getBatchCount()` → uint256
  - `hashMessage(string source, string sender, string text, uint256 timestamp)` → bytes32 (pure helper)
- **Events**: `BatchCommitted(batchId, merkleRoot, timestamp, messageCount, committer)`, `BatchVerified(batchId, leaf, valid)`
- **Privacy**: Only 32-byte Merkle roots stored on-chain, never message content

### Merkle Tree Generation (backend/merkle.py)
- Each message → leaf = SHA-256 of `"source|sender|text|timestamp"` (piped format)
- Leaves sorted, built into binary Merkle tree
- Functions: `hash_message()`, `build_merkle_tree()`, `get_merkle_root()`, `get_merkle_proof()`
- Proofs enable off-chain verification of message inclusion without revealing other messages

### Backend Vault Router (backend/routers/vault.py)
- `POST /api/vault/commit` — queries recent important messages (score >= threshold), builds Merkle tree, submits to Monad via backend wallet (configurable private key). Returns batch ID + tx hash + explorer URL.
- `GET /api/vault/batches` — lists all committed batches from the contract via `getBatchCount()` + `getBatch()` calls
- `GET /api/vault/batches/{id}` — single batch detail
- `POST /api/vault/verify` — verify a message proof against a committed root on-chain
- Uses lazy web3.py imports to avoid slowing server startup (web3 imports take ~7 seconds)

### Deployment
- **Hardhat**: `contracts/hardhat.config.js` + `ignition/modules/MemoryVault.ts` for `hardhat-ignition` deployment
- **Python**: `contracts/scripts/deploy.py` — standalone deploy script using solc + web3, handles compilation + deployment + verification in one command
- **Verification**: Explorer API integration for source code verification
- **Config**: Set `CONTRACT_ADDRESS` and `WALLET_PRIVATE_KEY` in .env after deployment

### Frontend — Memory Vault Panel
- Visible after wallet connect (hidden for password login)
- **Network switcher**: Dropdown (Monad Testnet / Monad Mainnet) calls `wallet_switchEthereumChain` / `wallet_addEthereumChain`
- **Commit Batch button**: POST /api/vault/commit → shows tx hash + explorer link
- **Batch list**: Renders all batches from contract with root preview + message count + explorer link
- **Commit status**: Shows "generating merkle tree..." → "batch #3 committed — view tx" with green link

### Why Monad for This (from README)
1. **Proof of existence without bloat**: Only 32-byte roots, not messages. Monad's low gas makes frequent commits practical.
2. **Parallel execution**: Multiple concurrent batch commits without contention
3. **Asynchronous deferred state**: Time-sensitive commits don't stall on prior state reads
